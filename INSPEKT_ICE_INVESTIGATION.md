# Inspekt Internal Compiler Error (ICE) Investigation - Deep Dive

## Root Cause Analysis

The ICE `org.jetbrains.kotlin.ir.declarations.impl.IrFunctionImpl has no callableId` is caused by a fragile implementation in `dev.rnett.inspekt.ir.passes.ReplaceInspektCalls` which relies on a strict, non-nullable implementation of
`callableId`.

### Technical Breakdown

The transformer overrides `visitCall` and attempts to access the `callableId` property of the target function for **every single function call** in the IR tree:

```kotlin
// dev.rnett.inspekt.ir.passes.ReplaceInspektCalls.kt
override fun visitCall(expression: IrCall): IrExpression {
    // This access triggers the crash if owner has no callableId
    val result = if (expression.symbol.owner.callableId == Names.inspect.asCallableId()) {
        intrinsify(expression)
    } else {
        expression
    }
    return super.visitExpression(result)
}
```

### The Crashing Property

The `callableId` property being accessed is an extension in `org.jetbrains.kotlin.ir.util.AdditionalIrUtils`. In Kotlin `2.x` (specifically seen in `2.3.10-sources`), this property is implemented as:

```kotlin
// org.jetbrains.kotlin.ir.util.AdditionalIrUtils.kt
val IrFunction.callableId: CallableId
get() = callableIdImpl

private val IrDeclarationWithName.callableIdImpl: CallableId
get() {
    if (this.symbol is IrClassifierSymbol) error("Classifiers can not have callableId. Got $this")
    return when (val parent = this.parent) {
        is IrClass -> parent.classId?.let { CallableId(it, name) }
        is IrPackageFragment -> CallableId(parent.packageFqName, name)
        else -> null
    } ?: error("$this has no callableId") // <--- CRASH HERE (line 105 in 2.3.10)
}
```

This implementation is **non-nullable** and explicitly throws an `IllegalStateException` (via `error()`) if a `callableId` cannot be determined (e.g., if the parent is an `IrFunction` for local functions, or if it's a synthetic function not
yet fully linked).

### Interaction with `kotlinx-serialization`

The `kotlinx-serialization` compiler plugin generates many synthetic functions and properties (e.g., `$serializer`, `write$Self`, `descriptor`).

1. Serialization generates these elements during its own IR pass.
2. `inspekt`'s IR pass runs and traverses the entire tree, including code generated by serialization.
3. When `inspekt` visits a call to one of these synthetic functions, it calls `.callableId`.
4. Because these functions often have unconventional parentage or are in a transient state, the compiler's strict `callableId` check fails and crashes the entire compilation.

## Environment

- **Kotlin:** 2.3.10 (seen in compiler-embeddable sources)
- **Inspekt:** 0.0.1
- **Kotlinx Serialization:** 1.10.0 (used in `easy-mcp`)

## Reproduction

The ICE is triggered when `inspekt` is applied to a module that also uses `kotlinx-serialization`, specifically in files with complex IR structures (like those using `Flow`, `Iterable`, and `@Serializable` types together).

**Consistently triggering file in `easy-mcp`:**
`core/src/test/kotlin/dev/rnett/easy/mcp/resource/ResourceReturnTypesIntegTest.kt`

## Proposed Fix for `inspekt`

The `ReplaceInspektCalls` transformer must avoid calling the strict `.callableId` property on every function call.

### Safe Alternatives:

1. **Direct Name/Package Check:**
   ```kotlin
   val owner = expression.symbol.owner
   if (owner is IrSimpleFunction && 
       owner.name.asString() == "inspekt" && 
       owner.getPackageFragment()?.packageFqName?.asString() == "dev.rnett.inspekt") {
       return intrinsify(expression)
   }
   ```

2. **Defensive CallableId Check:**
   ```kotlin
   val owner = expression.symbol.owner
   // Only check callableId if we know it won't crash (has a valid parent)
   val hasValidParent = owner.parent is IrClass || owner.parent is IrPackageFragment
   if (hasValidParent && owner.callableId == Names.inspect.asCallableId()) {
       return intrinsify(expression)
   }
   ```

## Workaround for `easy-mcp`

The current workaround is to isolate `inspekt()` calls or simplify the service structures in tests to avoid IR patterns that serialization and `inspekt` both struggle to process simultaneously.
